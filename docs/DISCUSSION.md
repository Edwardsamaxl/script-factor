# 问题讨论记录

## 1. 女娲 skill 分析

### 1.1 女娲是什么
女娲（nuwa-skill）是一个人物认知框架蒸馏工具，核心思路：
- 输入一个人名 → 6路并行 web search 调研 → 提炼心智模型+表达DNA+决策规则 → 输出可运行的 SKILL.md

### 1.2 核心处理流程
```
用户输入名字
    ↓
Phase 0.5 创建目录结构
    ↓
Phase 1 启动6个并行Agent（各自调用 web_search）
  Agent1: 著作搜索
  Agent2: 对话/访谈搜索
  Agent3: 表达风格搜索
  Agent4: 他者视角搜索
  Agent5: 决策记录搜索
  Agent6: 时间线搜索
    ↓
Phase 2 提炼（主Agent读取调研文件）
    ↓
Phase 3 组装 SKILL.md
    ↓
Phase 4 验证 + Phase 5 双Agent精炼
```

**关键点**：web_search 发生在**使用 skill 的大模型内部**，skill 本身只是一个**提示词框架**，告诉大模型"应该怎么调研、提炼、组装"。

### 1.3 女娲的5层提取内容

| 层次 | 对应人设构建 |
|------|------------|
| 怎么说话 | 表达DNA（语气、用词、节奏） |
| 怎么想 | 心智模型（认知框架） |
| 怎么判断 | 决策启发式 |
| 什么不做 | 反模式/价值观底线 |
| 知道局限 | 诚实边界 |

## 2. 一级创作者：用户如何把 prompt 变成人设

### 2.1 女娲方式 vs 我们需要的方式

| 场景 | 女娲 | 我们（一级创作者） |
|---|---|---|
| 输入 | 人名（网上搜） | 用户提供的素材 |
| 素材来源 | 6路 web search | 用户上传/录制 |
| 处理方式 | 相同的方法论 | 相同的方法论 |

### 2.2 用户可提供的素材形式

| 素材类型 | 处理方式 |
|---------|---------|
| 自己写的文章/帖子 | 直接分析 |
| 录音/视频（转 transcript） | 分析问答模式和即兴反应 |
| 填写的结构化问卷 | 更结构化，提取效率高 |
| 聊天记录整理 | 分析真实对话风格 |

### 2.3 一级创作者人设构建流程（借鉴女娲）

```
用户上传素材（文章/录音/问卷等）
    ↓
Phase 1 信息采集
  - 分析素材内容
  - 提取：表达风格、心智模型、核心观点
    ↓
Phase 2 框架提炼
  - 心智模型（2-4个）
  - 表达DNA（语气、用词、句式）
  - 决策规则/观点立场
    ↓
Phase 3 构建人设配置
  - 生成结构化 Persona 对象
  - 输出可存储的 JSON 配置
```

## 3. Agent 对话调用架构讨论

### 3.1 核心问题
调用 agent 进行对话是否需要服务器作为传话中介？还是让 agent 在后台生成完消息再传回服务器？

### 3.2 方案对比

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **服务器作为中介** | 前端 → 服务器 → LLM API → 服务器 → 前端 | 可记录日志、可干预、可缓存 | 增加延迟、需要后端服务 |
| **纯前端直调** | 前端 → LLM API 直连 | 简单、低延迟、无服务器成本 | 暴露 API key、安全风险 |
| **Serverless 中间层** | 前端 → Serverless 函数 → LLM API | 无服务器运维、按需扩展 | 冷启动延迟、费用 |
| **后台任务模式** | 前端发起 → 后台生成 → 轮询/推送拿结果 | 用户体验好、可处理长任务 | 架构复杂、需要状态管理 |

### 3.3 推荐方案：后台任务 + 轮询/Server-Sent Events

**原因**：
- 两个人设 agent 对话生成剧本耗时可能长达 10+ 分钟
- 不能让前端一直等待（HTTP 超时、用户体验差）
- 需要支持现场演示（预生成 + 异步展示）

**架构示意**：
```
前端发起生成请求
    ↓
服务器创建任务队列
    ↓
后台 worker 执行：
  1. 调用 LLM API（人设A）
  2. 收集回复
  3. 调用 LLM API（人设B）
  4. 收集回复
  ... 循环 N 轮
    ↓
任务完成，存储结果
    ↓
前端轮询 / SSE 收到完成通知
    ↓
展示剧本结果
```

### 3.4 对我们项目的建议

MVP 阶段可以用**纯前端直调**快速验证：
- API key 放在前端（风险可控，因为是 MVP）
- 单次剧本生成用单次 LLM 调用完成（不是真正的 multi-agent 对话）
- 把"两个人设对话"作为 **prompt 结构** 而不是真正的两轮 agent

后期切换到**后台任务模式**：
- Vercel Serverless Functions / 腾讯云 SCF
- 任务队列 + 轮询获取结果
- 支持真正的两个人设 agent 多轮对话

## 4. 现场演示可行性分析

### 4.1 问题
- agent 之间的对话有延迟，生成一个剧本可能要十几分钟
- 现场演示基本不可行

### 4.2 解决方案

**方案1：拆解演示，分步展示**
- 只演示"输入素材 → 人设构建"流程（3-5分钟）
- 剧本生成做成异步，结果预生成好，现场只展示输出

**方案2：减少对话轮次**
- 不是真正跑两轮 agent 对话
- 而是"场景设定 + 单 agent 生成剧本"
- 把"两个人设对话"作为 prompt 结构

**方案3：预生成 + 现场对比**
- 提前生成好几组剧本
- 现场从预生成结果"抽取"展示

**方案4：只做 agent 对话可视化（推荐）**
- 不等剧本生成完成
- 展示两个人设 agent 的**实时对话过程**（流式输出）
- 观众看到对话进行就觉得是实时演示
- 剧本生成放到后台

## 6. 方案决策（已确认）

### 6.1 Agent 对话：采用方案B（真对话）

```
Agent A: "xxx"  →  发给 B
Agent B: "yyy"  →  发给 A
Agent A: "zzz"  →  发给 B
... 循环 N 轮
```

**实现方式**：
- 每轮一次 API 调用
- 调用内容包含完整对话历史 + 当前轮次角色
- 返回该角色的发言
- 需要管理：对话历史、轮次控制、中间结果存储

**技术难度**：不高，就是标准的对话状态管理

### 6.2 演示策略：预生成 + 结果展示

- Agent 对话生成放后台跑（耗时 5-10 分钟）
- 提前生成一批效果好的剧本
- 现场演示直接展示结果 + 人设创建功能
- 不需要在现场等待生成过程

### 6.3 架构选择

MVP 阶段可以用**纯前端直调 + 本地存储**，快速验证：
- 单次 LLM 调用测试对话效果
- 对话历史存储在 localStorage
- 预生成剧本存储展示

后期切换到后台任务模式（Serverless）
